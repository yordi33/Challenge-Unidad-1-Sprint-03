01 — fundamentos

¿Qué es Git?

Git es un sistema de control de versiones distribuido creado por Linus Torvalds en 2005. Permite registrar el historial completo de cambios de un proyecto, trabajar en paralelo sin pisar el trabajo de otros, y volver a cualquier punto anterior.

Git ≠ GitHubGit es la herramienta que corre en tu máquina. GitHub, GitLab y Bitbucket son plataformas en la nube que alojan repositorios Git y añaden herramientas de colaboración.

Repositorio

Un repositorio (o "repo") es la carpeta que Git vigila. Contiene todos los archivos del proyecto y la carpeta oculta .git/ con toda la historia. Cada repositorio es independiente y puede vivir solo en tu máquina o sincronizarse con un remoto (GitHub).

Las tres zonas de Git

Directorio de trabajo ──→ Staging Area ──→ Repositorio (.git) (tus archivos) git add git commit
02 — flujo fundamental

Flujo de trabajo

Staging area

La staging area es un espacio intermedio donde preparas los cambios antes de confirmarlos. Puedes tener 10 archivos modificados y hacer commit solo de 3.

git add archivo.txt          # añade un archivo concreto
git add .                    # añade todos los cambios
git add -p                   # modo interactivo: elige qué partes añadir

Commit

Un commit es una instantánea permanente del proyecto. Cada commit tiene un hash SHA-1 único, un mensaje, el autor, la fecha, y un puntero al commit anterior.

git commit -m "feat: add user login form"
git commit                   # abre el editor para un mensaje largo
git commit --amend           # modifica el último commit (¡solo antes de push!)

HEAD

HEAD es un puntero que indica dónde estás ahora en la historia del repositorio. Normalmente apunta a la rama activa.

commit C ← commit B ← commit A ↑ HEAD (main)
03 — primeros pasos

Primeros comandos

Configuración inicial

git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"
git config --global core.editor "code --wait"
git config --list                   # ver toda la configuración

Crear e inspeccionar un repositorio

git init                    # inicia un repo en la carpeta actual
git status                  # estado: modified, staged, untracked
git add .                   # añade todo al staging
git commit -m "mensaje"     # guarda un commit

.gitignore

El archivo .gitignore indica a Git qué archivos nunca debe rastrear. Es imprescindible en cualquier proyecto real.

# .gitignore
node_modules/
.env
*.log
dist/
.DS_Store
__pycache__/
*.pyc
TipCrea tu .gitignore desde el inicio. gitignore.io genera uno automáticamente según tu stack.
04 — inspección

Historial y diferencias

Ver el historial

git log                          # historial completo
git log --oneline                # una línea por commit
git log --oneline --graph --all  # árbol visual de todas las ramas
git log --author="Nombre"        # filtrar por autor
git log --since="2024-01-01"    # filtrar por fecha

Ver diferencias

git diff                         # cambios sin stage
git diff --staged                # cambios en staging (listos para commit)
git diff main..develop           # diferencia entre dos ramas
git show abc1234                 # ver un commit concreto por su hash

Deshacer cambios

git restore archivo.txt          # descarta cambios (¡irreversible!)
git restore --staged archivo.txt # saca del staging sin borrar cambios
git revert abc1234               # deshace commit creando uno nuevo (seguro)
git reset --soft HEAD~1          # deshace commit, mantiene cambios en staging
Cuidadogit reset --hard borra cambios permanentemente. Úsalo solo si sabes exactamente lo que haces.
05 — ramas

Ramas (Branches)

Las ramas son líneas de desarrollo paralelas. Permiten trabajar en una nueva funcionalidad sin tocar el código estable de main.

main: A ── B ── C ──────────── F \ / feature: C ── D ── E

Comandos de ramas

git branch                       # lista ramas locales
git branch -a                    # lista locales + remotas
git branch nueva-rama            # crea una rama (no cambia a ella)

git switch nombre-rama           # cambia a una rama existente
git switch -c nueva-rama         # crea Y cambia de una vez ✓

git branch -d rama               # elimina (solo si ya fue fusionada)
git branch -D rama               # fuerza la eliminación

Convención de nombres

TipoEjemplo
Nueva funcionalidadfeature/user-login
Corrección de bugfix/navbar-overflow
Corrección urgente en producciónhotfix/payment-crash
Preparación de versiónrelease/v2.1.0
Experimentoexperiment/new-animation
06 — fusión

Fusión de ramas (Merge)

Fusionar integra los cambios de una rama en otra. Siempre te fusionas hacia la rama activa (HEAD).

git switch main              # primero ve a la rama destino
git merge feature/login      # trae los cambios a main

Fast-forward

Cuando la rama destino no tiene commits nuevos desde la bifurcación, Git simplemente mueve el puntero.

# Antes: main: A ── B \ feature: B ── C ── D # Después: main: A ── B ── C ── D ← solo movió el puntero

Merge commit

Cuando ambas ramas tienen commits distintos, Git crea un commit nuevo que une ambas historias.

# Antes: main: A ── B ── E \ feature: B ── C ── D # Después: main: A ── B ── E ── M ← M es el merge commit \ / feature: B ── C ── D
git merge --no-ff feature/login  # fuerza merge commit (recomendado en equipos)
git merge --squash feature/login # aplasta todos los commits en uno
07 — conflictos

Conflictos y stash

Un conflicto ocurre cuando dos ramas modifican las mismas líneas. Git te pide que decidas cuál versión prevalece.

Cómo se ve un conflicto

<<<<<<< HEAD
Esta es la versión de tu rama actual (main)
=======
Esta es la versión de la rama que estás fusionando (feature/x)
>>>>>>> feature/x

Pasos para resolver un conflicto

# 1. Intenta el merge
git merge feature/x

# 2. Git te avisa: CONFLICT (content): Merge conflict in archivo.txt

# 3. Abre el archivo, edítalo manualmente y elimina los marcadores

# 4. Marca el conflicto como resuelto
git add archivo.txt

# 5. Completa el merge
git commit
TipVS Code tiene un editor visual de conflictos ("Accept Current", "Accept Incoming", "Accept Both"). Es mucho más cómodo que editar el texto a mano.

Git stash — guardar trabajo sin hacer commit

git stash                    # guarda los cambios temporalmente
git switch otra-rama         # trabaja en otra cosa
git switch mi-rama           # vuelve
git stash pop                # recupera los cambios guardados
git stash list               # lista todos los stash guardados

Trabajo remoto

Conectar y clonar

git remote add origin https://github.com/usuario/repo.git

git push                                # si ya hay seguimiento configurado

09 — colaboración

GitHub y colaboración real

Pull Request (PR)

Un Pull Request es una propuesta de cambio: pides que tus commits en una rama se fusionen en otra. Es el mecanismo central de la colaboración.

  1. Creas una rama y trabajas en ella
  2. Subes los commits a GitHub
  3. Abres un PR en GitHub (base: main ← compare: feature/x)
  4. Un compañero revisa el código y comenta
  5. Corriges si hace falta y el PR se aprueba
  6. Se hace merge a la rama principal

Fork

Un fork es una copia completa de un repositorio en tu cuenta de GitHub. Permite contribuir a proyectos en los que no tienes permisos de escritura directa.

# 1. Fork en GitHub (botón en la web)
# 2. Clona tu fork
git clone https://github.com/TU_USUARIO/repo-forkeado.git

# 3. Añade el repo original como remoto "upstream"
git remote add upstream https://github.com/AUTOR_ORIGINAL/repo.git

# 4. Trabaja en una rama, haz push a tu fork
git switch -c mi-contribucion
git push origin mi-contribucion

# 5. Abre PR desde tu fork al repo original

10 — flujo simplificado

GitHub Flow — flujo simplificado

El GitHub Flow es el flujo recomendado para proyectos pequeños, equipos ágiles y entrega continua. Es el que usáis en los Team Challenges.

Cuándo usarloProyectos de bootcamp, equipos pequeños (<10 personas), proyectos sin ciclos de versión formal, entornos con CI/CD donde cada merge a main se despliega automáticamente.

Estructura de ramas

main: ●──────────────────────●──────────────────● \ ↗ \ ↗ feature/login: ●──●──●(PR) \ / \ / feature/dashboard: ●──●──●(PR)──

Reglas del GitHub Flow

  • main siempre contiene código que funciona y se podría desplegar
  • Todo trabajo nuevo va en una rama feature/*
  • Los cambios entran a main solo via Pull Request con revisión
  • No existe develop, ni hotfix, ni release

Flujo completo de una feature

# 1. Partir de main actualizado
git switch main
git pull origin main

# 2. Crear rama de trabajo
git switch -c feature/nueva-funcionalidad

# 3. Desarrollar con commits descriptivos
git add .
git commit -m "feat: add user login form"

# 4. Subir la rama a GitHub
git push -u origin feature/nueva-funcionalidad

# 5. Abrir Pull Request en GitHub
#    base: main ← compare: feature/nueva-funcionalidad

# 6. Tras el merge, limpiar
git switch main
git pull origin main
git branch -d feature/nueva-funcionalidad
11 — flujo completo

Git Flow — flujo completo

El Git Flow añade capas de estructura para proyectos con versiones formales, ciclos de release y equipos más grandes.

Cuándo usarloSoftware con versiones numeradas (v1.0.0, v2.3.1), equipos medianos/grandes, proyectos que no se despliegan continuamente sino por releases planificadas.

Estructura de ramas

main: ●──────────────────────────────────●────● \ ↗ ↗ develop: ●──●──────────────────●──●──● | \ \ ↗ ↑ | feature/A: ●──● \ / | | feature/B: ●──●──●──● release/ ●──● hotfix: (desde main) ●──●──────────/

Ramas permanentes

RamaPropósito
mainCódigo en producción. Solo recibe merges via PR desde release/* o hotfix/*.
developIntegración continua de features. Se despliega a entorno de staging.

Ramas temporales

TipoDesdeFusionar aPropósito
feature/*developdevelopNueva funcionalidad
fix/*developdevelopCorrección de bug en develop
release/*developmain + develop + tagPreparar un lanzamiento
hotfix/*mainmain + develop + tagCorrección urgente en producción

Flujo típico de una feature

# Partir de develop
git switch develop && git pull origin develop
git switch -c feature/nueva-funcionalidad

# Desarrollar...
git commit -m "feat: add feature"

# PR: feature/nueva-funcionalidad → develop
git push origin feature/nueva-funcionalidad

Flujo de un hotfix

# Partir de main (problema en producción)
git switch main && git pull
git switch -c hotfix/login-crash

# Arreglar y hacer commit
git commit -m "fix: resolve login crash on expired tokens"

# PR a main Y también a develop
git switch main && git merge hotfix/login-crash
git tag -a v1.0.1 -m "hotfix: login crash"

git switch develop && git merge hotfix/login-crash
git branch -d hotfix/login-crash

Tags: marcar versiones

git tag -a v1.0.0 -m "Primera versión"  # tag anotado (recomendado)
git push origin v1.0.0                   # subir un tag concreto
git push origin --tags                   # subir todos los tags
GitHub Flow — simplificado
  • Ramas permanentes: solo main
  • Ramas temporales: feature/*
  • PRs directamente a main
  • Sin develop, sin hotfix, sin release
  • Ideal para: bootcamp, proyectos pequeños, CI/CD
Git Flow — completo
  • Ramas permanentes: main + develop
  • Ramas temporales: feature/*, release/*, hotfix/*
  • PRs feature → develop, release → main
  • Hotfixes aplican a main Y develop
  • Ideal para: software versionado, equipos medianos
12 — best practices

Buenas prácticas

Conventional Commits

El estándar Conventional Commits da formato consistente a los mensajes y permite generar CHANGELOGs automáticamente.

<tipo>(ámbito opcional): <descripción corta en presente e imperativo>

[cuerpo opcional: el por qué, no el cómo]
TipoCuándo usarlo
featNueva funcionalidad
fixCorrección de bug
choreBuild, dependencias, configuración
docsSolo documentación
testAñadir o modificar tests
refactorRefactorización sin cambio de comportamiento
styleFormato, espacios (sin cambio de lógica)
perfMejoras de rendimiento

Ejemplos reales

git commit -m "feat(auth): add OAuth2 login with Google"
git commit -m "fix(cart): prevent negative item quantity"
git commit -m "chore: upgrade dependencies to latest"
git commit -m "docs: update README with setup instructions"
git commit -m "feat!: redesign user profile API"  # ! = breaking change

Commits atómicos

Cada commit debe representar un solo cambio lógico. Un buen commit es pequeño y enfocado, el código funciona antes y después, y el mensaje explica el qué y el por qué, no el cómo.

Checklist antes de hacer push

  • git status limpio (nada inesperado)
  • git diff --staged revisado
  • Mensaje de commit en formato Conventional Commits
  • .gitignore actualizado si añadiste archivos ignorables
  • Tests pasando (si los hay)
13 — práctica

Ejercicios

Ejercicio 1

Tu primer repositorio

  1. Crea una carpeta mi-proyecto y entra en ella.
  2. Ejecuta git init y verifica con git status.
  3. Crea un archivo .gitignore con *.log dentro.
  4. Crea README.md con una línea de texto.
  5. Añade ambos al staging con git add .
  6. Haz el primer commit: git commit -m "chore: initial project setup"
  7. Verifica el resultado con git log
Ejercicio 2

Ramas Patronus

  1. Crea una carpeta Patronus e inicializa un repositorio Git.
  2. Crea patronus.txt vacío y haz commit: "chore: add empty patronus file"
  3. Crea dos ramas desde main: harry y snape.
  4. En rama harry: añade arte ASCII del ciervo. Commit: "feat: add Harry's stag patronus"
  5. En rama snape: añade arte ASCII de la cierva. Commit: "feat: add Snape's doe patronus"
  6. Desde snape, crea la rama lily con git switch -c lily.
  7. Ejecuta git log --oneline --graph --all para ver el árbol.
  8. Bonus: elimina la rama snape con git branch -d snape.
Ejercicio 3

Fusionando el Patronus

  1. Cambia a main y fusiona la rama harry.
  2. Fusiona también la rama lily. Observa el merge commit.
  3. Visualiza la historia con git log --oneline --graph --all.
  4. Elimina las ramas ya fusionadas: git branch -d harry lily.
Ejercicio 4

Conectar con GitHub

  1. Crea un repositorio nuevo en github.com (sin inicializarlo).
  2. En tu repo local, conecta el remoto: git remote add origin https://github.com/TU_USUARIO/mi-proyecto.git
  3. Sube tu trabajo: git push -u origin main
  4. Haz un cambio en README.md directamente en GitHub (edita en la web).
  5. Descarga el cambio con git pull y verifica que está en tu máquina.
Ejercicio 5 — integrador

Flujo completo en equipo (GitHub Flow)

  1. Crea un repo tienda-online con README.md inicial.
  2. Desde main, crea feature/productos.
  3. Añade productos.txt con 3 productos. Haz 2 commits con mensajes Conventional Commits.
  4. Sube y abre un PR en GitHub: feature/productos → main.
  5. Desde main, crea feature/carrito. Añade carrito.txt. Haz commit.
  6. Abre un PR para feature/carrito también.
  7. Fusiona ambos PRs y visualiza toda la historia: git log --oneline --graph --all.
